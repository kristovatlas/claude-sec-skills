# Threat Model Generation Skill

## When to Use

Use this skill when the user asks you to generate a threat model, security analysis, or STRIDE analysis for an application or system you have access to in the current codebase.

## Prerequisites

Before generating the threat model, you MUST thoroughly explore the codebase to understand:

1. **Architecture**: Runtime environment, hosting, databases, external services, deployment pipeline
2. **Entry points**: APIs, webhooks, user interfaces, CLI commands, scheduled jobs
3. **Data flows**: What data enters the system, where it's stored, where it's sent, what transformations occur
4. **Authentication & authorization**: How users/services authenticate, permission models, token handling
5. **Integrations**: Third-party APIs, SDKs, platform-specific handlers, OAuth flows
6. **Dependencies**: Package manifests (`package.json`, `requirements.txt`, `go.mod`, etc.), notable transitive dependencies
7. **Secrets management**: How API keys, tokens, and credentials are stored and accessed
8. **Configuration**: Environment variables, config files, feature flags

Read all relevant source files. Do not guess or assume — base your analysis on actual code.

## Model to Use

When generating the threat model content, use `claude-opus-4-6` via the Anthropic API or subagent if available. The analysis requires the highest quality reasoning. If you cannot delegate to Opus, proceed with your best analysis but note the model used.

## Handling Existing Threat Models

If a `THREAT_MODEL.md` (or equivalent) already exists in the repo, **never assume it is complete**. Whether the file was written by a human or generated by a previous agent run, treat it as potentially incomplete and review it against the current codebase.

1. **Read the existing file in full.** Understand what's already documented — assets, threats, countermeasures, STRIDE/LINDDUN analyses, and their cross-references.
2. **Compare against the codebase.** Perform the same thorough codebase exploration described in Prerequisites. Identify any gaps: assets not covered, threats not considered, countermeasures missing or with outdated status, STRIDE/LINDDUN categories left empty or stale, etc.
3. **Fill gaps where they exist.** Add missing assets, threats, countermeasures, or analysis. Follow the same ID conventions already in use in the file (e.g., if the last asset is A7, new assets start at A8). Apply **(New — [Month Year])** markers to additions.
4. **Do not add content for the sake of completeness.** If the existing model already adequately covers an area, leave it alone. Every addition must be justified by a genuine gap — an unaddressed attack surface, an undocumented data flow, a missing countermeasure for an existing threat, or a LINDDUN/STRIDE category that should have entries but doesn't. Do not pad sections, inflate asset counts, or manufacture low-value threats just to make the document longer.
5. **Do not remove or rewrite existing content** unless it is factually incorrect based on the current codebase (e.g., references a component that no longer exists). If content is outdated, follow the incremental update conventions (strikethrough for retired items, update annotations with dates).
6. **Validate coverage constraints** after your review: every asset has at least one threat, every threat has at least one countermeasure, and the countermeasure status table reflects the current codebase.

## Output Format

Generate a single Markdown file named `THREAT_MODEL.md` (or a name the user specifies) in the repository root (or location the user specifies). Follow the template below exactly, adapting content to the specific application.

## Analysis Guidelines

When filling in the template, follow these principles:

### Assets
- The number of assets should scale with the complexity of the application. A simple single-purpose service might have 5-10 assets; a complex multi-service system with many integrations could have up to 50. Identify as many distinct assets as the system warrants — do not artificially constrain or inflate the count.
- **Every asset must be referenced by at least one threat.** If you identify an asset but cannot articulate a credible threat against it, reconsider whether it belongs in the model.
- Always consider: user data / PII, database contents, source code, authentication credentials / API keys, service identity and trust relationships, availability of critical workflows, third-party integration access, and any domain-specific sensitive data (financial records, health data, etc.).
- Rate values honestly. Not everything is High. A well-calibrated model has a mix of Low, Medium, and High ratings with clear justifications.
- Examples of assets include: database records, source code, non-financial PII (email address, dob, etc.), financial PII (identifying information combined with financial data such as bank account numbers or crypto-currency addresses), session tokens, API keys, user funds, site availability, encryption keys, DNS infrastructure (control over records or registrar), etc.

### Threats
- **Every threat must have at least one countermeasure.** If you identify a threat with no feasible countermeasure, note it explicitly as an accepted risk and explain why.
- **Every asset must be targeted by at least one threat.** If an asset has no threat, either the asset is not worth listing or you have not considered all attack vectors.
- Each threat must be specific to the application — generic threats like "an attacker could hack the server" are not acceptable.
- Ground every threat in something concrete: a code pattern, an architectural choice, a dependency, a configuration, or a data flow you actually observed.
- Likelihood should consider: Is the attack surface internet-facing? Is the data valuable? Are there known exploit patterns for the technologies used? Would an attacker be motivated to target this specifically?
- Mark threats introduced by proposed changes with **(New)**.
- Include privacy-specific threats informed by the LINDDUN analysis where they are actionable (e.g., excessive data retention, lack of user consent mechanisms, unnecessary linkability across platforms).

### Countermeasures
- **Every threat must have at least one countermeasure.** Most countermeasures mitigate multiple threats.
- Be specific and actionable. Reference actual files, functions, libraries, or configurations in the codebase.
- Include both preventive (stop the attack) and detective (notice the attack) countermeasures where appropriate.
- Suggest specific tools or libraries where relevant (e.g., `socket.dev` for dependency scanning, `helmet` for Express headers).
- **Verify implementation status.** For each countermeasure, search the codebase for evidence that it has been applied. This is one of the most valuable outputs of the threat model — a clear picture of what's protected and what's exposed.

### LINDDUN Analysis
- LINDDUN should be informed by the same codebase exploration as STRIDE but with a privacy lens. Focus on data flows involving personal data: where it enters, where it's stored, who can access it, where it's sent externally, and how long it's retained.
- Not every LINDDUN category will apply to every application. A backend service with no end-user PII may have minimal Identifiability or Unawareness concerns. Be honest about what applies and mark inapplicable categories as N/A with a brief justification.
- Where LINDDUN findings are actionable, they should generate corresponding entries in the Threats and Countermeasures sections (e.g., a Linkability finding might lead to a threat about cross-platform user correlation and a countermeasure recommending pseudonymization).
- Privacy threats can reference both the LINDDUN analysis and the STRIDE analysis — the two frameworks are complementary, not siloed.

### General Quality Standards
- **No generic filler.** Every sentence should convey information specific to the application being analyzed.
- **Cross-reference consistently.** Asset IDs (A1, A2...), Threat IDs (T1, T2...), and Countermeasure IDs (C1, C2...) must be used consistently across all tables and subsections.
- **Justify ratings.** Every Low/Medium/High rating needs a brief rationale grounded in the actual system.
- **Acknowledge uncertainty.** If you cannot determine something from the code alone (e.g., network configuration, cloud IAM policies), say so explicitly rather than guessing.
- **Consider the proposed changes.** If the user mentions planned features or you see feature branches, analyze how they change the threat landscape. Mark new assets, threats, and countermeasures with **(New)**.
- **Ensure complete coverage.** Every asset must have at least one threat. Every threat must have at least one countermeasure. Validate this as a final step before delivering the threat model.

## Incremental Update Workflow

When updating an existing threat model (e.g., after new features are merged, architecture changes, or periodic review), follow this workflow instead of regenerating from scratch.

### When to Use Incremental Updates

Use this workflow when:
- The user says something like "update the threat model," "re-run for the new auth service," "we merged the PR, update the threats," or "add [feature] to the threat model"
- An existing `THREAT_MODEL.md` (or equivalent) already exists in the repo
- The changes are scoped — not a full rewrite of the application

If the application has changed so fundamentally that >50% of the assets or threats would need rewriting, generate a fresh threat model instead and note that it supersedes the previous version.

### Step 1: Diff Analysis

Before modifying anything, understand what changed:

1. **Read the existing threat model** in full.
2. **Identify what changed in the codebase** since the model was written. Use the following signals:
   - Git history: `git log --since="[date of last threat model]" --oneline` or diff against a known baseline
   - User description of changes (if provided)
   - New files, deleted files, modified handlers/routes/services
   - Changes to `package.json`, `requirements.txt`, or other dependency manifests
   - Changes to environment variables, secrets configuration, or deployment configs
3. **Categorize the changes** into:
   - **New attack surface**: New endpoints, integrations, data stores, user roles, or external service calls
   - **Removed attack surface**: Deprecated features, removed endpoints, decommissioned integrations
   - **Modified attack surface**: Changed auth flows, updated data handling, refactored services

### Step 2: Update Assets

For each change identified:

- **New assets**: Assign the next available identifier (e.g., if A9 exists, start at A10). Add to the assets table and create a subsection. Mark with **(New — [Month Year])**.
- **Removed assets**: Do NOT delete them. Instead, strike through the row in the table and add a note in the subsection: `**Retired [Month Year]**: [Brief reason, e.g., "Discord integration was removed in v2.3."]` This preserves audit history.
- **Modified assets**: Update the description and re-evaluate the value rating. If the rating changes, note the previous rating: `Value changed from **Medium** to **High** ([Month Year]): [reason].`

### Step 3: Update Threats

- **New threats**: Assign the next available ID. Add to the threats table and create a subsection. Mark with **(New — [Month Year])**. Ensure every new threat references at least one asset and has at least one countermeasure.
- **Retired threats**: Strike through in the table and add to the subsection: `**Retired [Month Year]**: [reason].` Do not delete.
- **Modified threats**: Update description, re-evaluate Likelihood/Impact/Severity. If severity changes, note: `Severity changed from **Medium** to **High** ([Month Year]): [reason].`
- **Re-evaluate existing threats**: Even threats that aren't directly affected by the code change may shift. For example, adding a new public-facing endpoint increases the likelihood of DoS threats. Briefly review all existing threats and adjust if warranted.

### Step 4: Update Countermeasures

- **New countermeasures**: Assign the next available ID. Link to the threats they mitigate.
- **Retired countermeasures**: Strike through and note retirement reason (e.g., the threat they addressed no longer exists, or they were replaced by a stronger control).
- **Modified countermeasures**: Update implementation guidance to reflect current code paths, libraries, or configurations.
- **Re-evaluate status**: Check whether previously unapplied countermeasures have now been implemented, or whether applied countermeasures have been removed or broken by recent changes. Update the status table accordingly.

### Step 5: Update the STRIDE and LINDDUN Summaries

Re-examine both the STRIDE and LINDDUN analyses. Add new attack/privacy scenarios, remove scenarios that no longer apply (with strikethrough), and ensure every bullet still reflects the current state of the system. New data flows or integrations may introduce new LINDDUN concerns even if the STRIDE surface hasn't changed significantly.

### Step 6: Add a Change Log Entry

Append a `# Change Log` section at the bottom of the threat model (create it if it doesn't exist):

```markdown
# Change Log

| **Date** | **Author/Trigger** | **Summary of Changes** |
| --- | --- | --- |
| [Month Year] | [What prompted the update, e.g., "AI features merged (PR #142)"] | Added A10, A11. Added T7, T8. Updated T3 severity from Medium to High. Added C14, C15. Retired T2. |
| [Previous date] | [Previous trigger] | [Previous summary] |
```

### Step 7: Validate Cross-References

After all updates, verify:
- Every asset is referenced by at least one threat
- Every threat is referenced by at least one countermeasure
- Every countermeasure references at least one threat
- No orphaned IDs exist (referenced in tables but missing subsections, or vice versa)
- All **(New)** and **(Retired)** markers include the date
- Countermeasure status table is accurate against the current codebase

### Example: Incremental Update Annotation Style

When updating an existing asset subsection:

```markdown
## (A5) Slack Channel Contents

The bot can post in certain Slack channels like `#my-other-slack-channel`.

The bot reads channels like `#my-slack-channel` to generate AI summaries and posts them
to `#my-other-slack-channel`.

**Updated Oct 2025**: The bot now also reads `#some-other-channel`
for post-mortem summaries. This channel contains more sensitive operational
detail than the previously scraped channels. Value changed from **Low** to
**Medium** (Oct 2025): incident response discussions may include
infrastructure credentials and vendor contact details.
```

When retiring a threat:

```markdown
## (T2) ~~AI Prompt Injection~~

**Retired Nov 2025**: AI features were removed in v2.4 after cost-benefit
review. The OpenRouter integration and all AI summarization code have been
deleted. This threat no longer applies.
```
